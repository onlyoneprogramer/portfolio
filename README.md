# 이진호 | 보안·시스템 관점에서 0→1을 만드는 소프트웨어 엔지니어

> **Top-down으로 사고하고, Bottom-up으로 검증하며,  
> 검증 결과를 구조로 고도화하는 엔지니어**

- **보안·운영 관점**에서 시스템을 설계하고, 실제 환경에서 “돌아가는 것”을 빠르게 검증합니다.
- **Top-down(구조·리스크·경계 정의) → Bottom-up(PoC 검증) → 운영 가능한 구조로 정리** 흐름으로 문제를 해결합니다.
- 하드웨어 연동/플랫폼/보안처럼 **문서만으로 정답이 나오지 않는 영역**에서 강점을 발휘합니다.

> 일부 프로젝트는 보안/대외비 이슈로 인해 제품명·코드·정확한 토폴로지는 비공개이며,  
> 공개 가능한 범위에서 **문제 해결 방식과 아키텍처 관점**을 중심으로 정리했습니다.

---

## 🧭 개발 방식 (How I Work)

### 1) Top-down 이해
- 요구사항·목표·제약 조건을 **시스템 단위로 정리**
- 공격자 관점(침해 시나리오)과 운영자 관점(장애·모니터링)을 동시에 고려
- 모듈 경계와 책임을 먼저 정의하고, **향후 무너질 수 있는 지점**을 예측

### 2) Bottom-up 구현 및 검증
- 가장 불확실한 요소부터 **PoC로 우선 검증**
- 커널·네트워크·센서·프로토콜·데이터 흐름을 직접 확인
- 검증 결과를 반영해 설계를 수정하고, 다시 확장 가능한 형태로 고도화

### 3) 운영 가능한 구조로 정리
- 장애 대응, 로그, 관측 가능성(Observability)을 구조에 포함
- 데이터 무결성, 예외 처리, 권한·접근 제어 등 **운영 리스크를 설계 단계에서 반영**

## 1) 웹 기반 로봇·센서 모니터링 시스템 (아키텍처 설계 및 구현)

**한 줄 요약**: 로봇/센서 실시간 이벤트 스트림을 수집·검증·가공하고, 운영/디버깅 가능한 웹 모니터링 구조로 재정리  
**기간**: 2025.11 ~ 2025.12  
**규모**: 로봇 **4대**, 로봇당 센서 **10개** → 총 **40개 센서**  
**특징**: **실시간 이벤트 스트림** 기반 모니터링

### 역할
- 웹 모니터링 시스템 **설계 및 구현 주도**
- 전체 소프트웨어 **아키텍처 정의**(레이어, 책임 경계, 데이터 흐름)
- 센서·로봇·에이전트·백엔드·웹 간 **데이터 계약(Contract) / 이벤트 흐름 설계**

### 문제
- 파트별 개발로 **End-to-End 구조 부재**
- 데이터 흐름 및 책임 경계 불명확 → **장애 원인 추적/대응 비효율**
- 실시간 처리와 이력 저장 로직 혼재 → **확장/운영/유지보수 난이도 증가**

### 해결
- 전체를 `Sensor → Edge/Agent → Backend API → Web Monitoring` 레이어로 정리해 **역할/책임 경계**를 명확화
- 실시간 이벤트에 대해 **수집 → 정합성/신뢰도 검증 → 표준 이벤트 포맷 변환** 파이프라인 정의
- 운영성과 확장성을 위해 **Hot Path(실시간)** / **Cold Path(이력)** 로직 분리
- 웹 모니터링에 **WebSocket 기반 실시간 상태 반영**을 적용하여 운영 화면에서 즉시 확인 가능하도록 구성

### 결과
- 혼란 상태의 구조를 **확장 가능한 아키텍처로 재정리**
- 로봇·센서·백엔드·웹을 하나의 흐름으로 이해할 수 있는 **공통 기준** 마련
- 모니터링 UI를 단순 관제가 아닌 **운영·디버깅 도구(상태 추적/원인 분석)**로 활용 가능하게 전환

**기술 스택**: Spring, WebSocket, PostgreSQL  
**핵심 키워드**: `레이어링` `Edge/Agent` `실시간 이벤트` `데이터 계약` `Hot/Cold 분리` `운영/디버깅`

## 2) ERP 시스템 개발 (단독 0→1 구축)

**기간**: 2024.01 ~ 2024.05  
**한 줄 요약**: 요구사항 변화가 잦은 ERP 도메인을 모듈화하고, 트랜잭션/쿼리 병목을 실측 기반으로 개선  
**역할**: 단독 개발 (요구사항 분석 → 설계 → 구현 → 배포)

### 문제
- 재고·주문·회계 요구사항이 복잡하고 변경이 잦음
- 성능 병목 및 데이터 일관성 문제 가능성 존재

### 해결
- 도메인 단위 분해로 비즈니스 로직을 모듈화하고 API/데이터 흐름을 정리
- 대용량 트랜잭션 병목 구간을 우선 식별 후 **쿼리·인덱스**를 실측 기반으로 반복 개선
- 실제 데이터 흐름을 기준으로 API 응답 구조를 조정하며 안정화

### 결과
- **모듈화된 엔터프라이즈 로직** 구축
- RESTful API 기반 데이터 흐름 일관성 확보
- 인덱스 최적화 및 쿼리 리팩토링으로 **응답 속도 40% 이상 개선**

**기술 스택**: Java, Spring, JSP/Servlet, MySQL
